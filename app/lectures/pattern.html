Web Programming Patterns

Mobile constraints


  • The network is unreliable;
  • File sizes need to be small;
  • Debugging requires logging;
  • Minimize user input;
  • Minimize time to result.

Architectural pattern


A classical telematic application architecture is shaped as following:

Three tier architecture


  • The presentation tier translates tasks and results in something the user can understand;
    it runs on the client side.
  • the logic tier executes commands and performs calculations;
  • the data tier stores the informations and pass them to the logic tier for processing.
    they run on the server side, sometimes on different systems.

Three tier architecture


The data tier is implemented by using an out-of-the-shelf database.

The logic and the presentation tiers are both implemented by the developer using frameworks.

Moreover there are standard solutions (patterns) used to write good, scalable and maintainable code.

A modern approach


  • Wide separation from client and server;
  • the logic tier is implemented on both client and server side;
  • commands and requests are sent by using the HTTP protocol.

Defining your data tier entities.


To define the data set we need to extract:

  • the entities acting a role in the context;
  • their attributes defined as atomic properties;
  • the relations existing between the entities.

Entity-Relationship model

The data architecture can be shaped using the UML language.

Implementing the logic tier


For both client and server side, the logic tier can be implemented by using an architectural pattern widely used.

It's name is Model-View-Controller.

Nowadays there are many frameworks that helps a developer to implement this patter for any device in any language.

Implementing the communication protocol


Client and server need a protocol (a set of rules) that defines the way the informations can be retrieved.

The most used communication pattern is REpresentational State Transfer (REST).

Representational state transfer


REST is a simple stateless architecture that generally runs over HTTP.

It is based on the Hypermedia as the Engine of Application State (HATEOAS) principle.

Hypermedia as the Engine of Application State


A client interacts with a network application entirely through hypermedia (URLs) provided dynamically by application servers.

Client and server don't need to share an interface anymore.

This allows the server functionalities to evolve independently.

REST in web services


RESTful APIs leverage on:

  • internet media types (JSON, XML, images...)
  • HTTP methods (GET, POST, PUT, DELETE)
  • hyperlinks to reference resources (http://example.com/item)

REST fundamentals


  • Nouns to define URLs
  • Two types of data:
    • Resources
    • Collections (Set of resources)
    Data are represented in XML or JSON format.

REST fundamentals


Data operations are mapped 1:1 over HTTP methods:

PUT => Create
GET => Read
POST => Update
DELETE => Delete

PUT and POST can both be used for Create and Update
